Давай составим план, как можно реализовать state-machine для нашего чата, что я хочу:
1. Создать отдельный мод для чата со state-machine (В приложении должен быть выбор при создании нового чата (standard или state-machine), в левой панеле чаты со state machine должны быть помечены иконкой мозга в начале с левой стороны айтема)
2. Состояния машины:
 i. Planing - планирование ответа/решения на постевленный вопрос/задачу от юзера. 
 В Planing могут ходить подзадачи, если пользователь изначально в промте не дал полного контекста:
Clarification - уточнение у пользователя доп. контекста исходя из составленного плана. Уточнения пользователя должны запоминаться и переформировывать/дополнять план
На Planing этапе модель не должна писать полноценный код, а лишь предоставить план дальнейшей реализации/решения исходя из запросов юзера.
 ii. Execution - реализация решения/ответа на вопрос пользователя согласно поставленному плану из состояния Planing. Может включать в себя любые методы формирования ответа.
 iii. Validation - сопоставляет решение/реализацию/ответ на этапе Execution с планом составленным на этапе Planing.
 iV. Done - итоговый результат по задаче (выводит Done, если все этапы завершились корректно, Canceled, если работа была отменена, Failed - с выводом причины, почему не удалось выполнить задачу). Статус выводится в чат
3. При нажатии на stop с этом режиме работа прекращается на этапах Planing, Execution, Validation, происходит переход в Done и выводится сообщение "Canceled"
4. Контекст берется и передается лишь в пределах одного [Planing] -> ... [Done], то есть, если задача завершена - при старте следующей [Planing] контекст предыдущих тессий не учитывается. Учитываются только description items из профиля, опции из chat context и текущая сессия [Planing] -> ... [Done]
5. Переход из planing в execution должен явно подтверждаться пользователем и сопровождаться сообщением "Перейти к выполнению плана?". До тех пор, пока пользователь не аппрувнет план, любые сообщения участвую в формировании/корректировке плана (значит отправляются в апи)
6. В Execution передается контекст лишь того плана, который по итогу был сформирован при аппруве пользователем.
7. После отработки Validation, если валидация успешная -> переход в done, если нет, то нужно явное уточнение от пользователя, что требуется сделать: 1. С учетом ответа на validation переделать решение из стадии execution -> переход на execution 2. Принять текущее решение -> переход в done )
8. Состояние сесии должно сохраняться между этапами и на подэтапах, чтобы при перезаходе в приложение или чат продолжать с того места где остановились (По кнопке "Продолжить выполнение"). Кнопка должна быть доступна и видима, только если оборвана/незакончена сессия и приложение было закрыто, но затем пользователь перезашел в приложение и заходчет продолжить сессию
9. Сверху чата должен высвечиваться список состояний [Planing] -> [Execution] -> [Validation] -> [Done]. Каждый этап должен быть помечен если он выполнен, в процессе выполнения или пока очередь не дошла до него соответствующими иконками)
10. Переход между стадиями регулируется кодом
11. Чат должен быть инактивен для ввода, пока от пользователя явно не потребуется ответ в текстовом виде
12. Если Planing или Clarification требует дополнения от пользователя, то пусть агент возвращает в конце "need context", по этому ключу будет парсится ответ агента и решать нужны ли уточнения у пользователя. Или "full context", если вопросов по плану больше нет.
13. На стадии Clarification у пользователя должна быть возможность по кнопке "Перейти к выполнению" - пропустить уточнения и перейти к execution с последним актуальным планом. В других статусах "Перейти к выполнению" не отображается
14. Ответ Validation в конце должен содержать Approve, если решение удовлетворяет плану или Decline, если есть отклонения. Эти ключи должен парсить код и решать про переход описаный в п.7
15. description items из профиля и опции из chat context должны расцениваться как инварианты и не давать пользователю изменить план или потребовать решение. То есть, если description items из профиля или опции из chat context содержат "Kotlin", а пользователь требует решение на c++, то агент должен отклонить требование пользователя ссылаясь на инвариант
16. Если чат со state-machine пустой, то это означает, что пользователь находится в состоянии [Planing] и может отправить первичный промпт
17. Подзадачи должны непосредственно влиять на статус основной заадчи. То есть, если подзадача находится на стадии выполения, то основная задача должна быть помечена соответсвующим статусом
18. Обрезай ключи статусов стадий (need context, full context и т.д.) после ответа модели, пользователю не нужно об этом знать
19. При переходе на стадию, она должна запускаться автоматически
20. Кнопки в панеле действия должны отображаться и быть доступны только для тех статусов, для которых они предназанчены. Вдругих статусах их не должно быть видно
21. При переходе между этапами - визуально помечай эти переходы в чате. (Переходы на и между подэтапапами не отображать)
22. При переходе к [Execution] из [Planing] в качестве user prompt должен подставлять итоговый план
23. При переходе к [Execution] из [Validation] в промпт должен подставляться итоговый план из [Planing], последний [Execution] и ответ из [Validation]
24. Интерактивные действия такие как "Перейти к выполнению" и т.д нужно перенести в чат на сторону вывода сообщений пользователя. После взаимодействия с кнопками/новым вводом пользователя и т.д. - кнопки скрывать из истории чата, то есть отображать только в тот момент, когда они подвержены взаимодействию
25. Переходы могут быть строго следующего вида, перепрыгивание этапов должно быть запрещено програмно (то есть сам код работает только по заданным сценариям):

Возможные переходы:
planing -> execution
planing -> clarification
clarification -> clarification
clarification -> execution
execution -> validation
validation -> execution
validation -> done
done -> planing (обунелние сессии, то есть если одна заадача закончена, пользовательн можешь приступить к следующей)
